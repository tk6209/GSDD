## PURPOSE

The GSDD system enforces a governed execution model where:

- NO action is permitted without an explicit and validated SPEC.
- Execution is a privilege, never a default state.

The objective of this law is to eliminate:

- implicit assumptions
- premature execution
- conversational authority
- uncontrolled scope expansion

---

## FOUNDATIONAL PRINCIPLES

### SPECIFICATION_FIRST

Execution MUST be preceded by a written SPEC.

If no SPEC exists:
- Execution is INVALID.
- Execution MUST NOT begin.

---

### EXPLICIT_OVER_IMPLICIT

Implicit context, inferred intent, or assumed requirements are FORBIDDEN.

All constraints, inputs, and success criteria MUST be explicitly declared.

---

### GOVERNANCE_EXECUTION_SEPARATION

Governance and execution are DISTINCT SYSTEM STATES.

- Governance evaluates, constrains, and authorizes.
- Execution performs ONLY what was authorized.

Execution while governance is unresolved is FORBIDDEN.

---

### DETERMINISM_OVER_CONVENIENCE

If ambiguity exists:
- The system MUST STOP.

Speed, convenience, or perceived simplicity NEVER justify execution.

---

### GOVERNANCE_BEFORE_OUTPUT

Producing output without prior governance validation is a FAILURE,
even if the output is technically correct.

---

### SINGLE_SCOPE_RULE

Each execution MUST operate on exactly ONE explicitly bounded scope.

Multi-scope execution is FORBIDDEN.

---

### EXECUTION_FINALITY

Execution is ATOMIC and TERMINAL.

Once execution begins:
- It MUST either COMPLETE successfully or ABORT.
- It MUST NOT self-correct, retry silently, or evolve.

There is NO concept of ongoing execution.

---

## VALID_EXECUTION_CONDITIONS

An execution is VALID ONLY IF ALL conditions below are met:

1. A SPEC exists and is accessible.
2. The SPEC defines exactly one bounded scope.
3. Inputs, constraints, and expected outcomes are explicit.
4. No ambiguity remains unresolved.
5. Execution was explicitly authorized after validation.
6. The outcome was audited against the SPEC before acceptance.

Failure of ANY condition invalidates execution.

---

## PROHIBITED_EXECUTION

The following are STRICTLY FORBIDDEN:

- Execution without a SPEC.
- Execution based on inferred or assumed intent.
- Execution across multiple scopes.
- Execution while ambiguity exists.
- Execution performed "to see what happens".
- Execution justified by urgency or convenience.
- Execution that mutates its governing SPEC.

Prohibited execution MUST be ABORTED immediately.

---

## ROLE_OF_THE_SPEC

The SPEC is the SINGLE SOURCE OF EXECUTION TRUTH.

The SPEC:
- defines scope
- defines boundaries
- defines success criteria
- constrains execution

Execution MUST conform to the SPEC.
The SPEC MUST NOT adapt to execution results.

---

## AMBIGUITY_HANDLING

Ambiguity includes, but is not limited to:

- missing information
- conflicting constraints
- unclear scope boundaries
- undefined success criteria

If ambiguity is detected:
- Execution MUST NOT proceed.
- Clarification MUST be requested.
- Speculative resolution is FORBIDDEN.

---

## OUTCOME_AUDIT

All execution outcomes MUST be audited against the SPEC.

- Audit is REQUIRED before acceptance.
- Audit failure FORBIDS commit.
- Subjective validation ("looks correct") is INVALID.

Audit is a GOVERNANCE REQUIREMENT.

---

## STOP_CONDITIONS

Execution MUST STOP immediately if:

- the SPEC is missing or incomplete
- scope boundaries become unclear
- new requirements emerge mid-execution
- constraints conflict with the SPEC
- execution deviates from declared intent

Stopping is a SUCCESSFUL outcome under GSDD.

---

## QUESTION_REQUIREMENT

Questions MUST be raised when:

- intent cannot be explicitly determined
- multiple valid interpretations exist
- execution choices affect scope or outcome
- constraints are underspecified

Questioning is a GOVERNANCE MECHANISM.

---

## ABORT_RULE

An ABORT MUST occur when:

- the SPEC cannot be validated
- ambiguity persists after clarification attempts
- execution would violate any foundational principle
- scope cannot be constrained safely

Aborted executions MUST NOT be partially completed.

---

## CORRECTION_AND_REEXECUTION

Any correction, fix, or improvement:

- MUST NOT modify an authorized execution
- MUST initiate a NEW execution cycle
- MUST require a NEW or UPDATED SPEC

Correction is NEVER continuation.

---

## AUTHORITY_AND_FINALITY

No execution may override this law.

No output, result, or perceived correctness justifies violation.

GOVERNANCE IS ABSOLUTE.
